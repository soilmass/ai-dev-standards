# Data Modeling

Rules for schema design that survive migrations, scale, and agent edits. The ORM and database are stack choices; these rules apply to whatever they are.

## Naming

1. Table/collection names: plural, snake_case (`user_accounts`); column/field names: snake_case, no prefixes repeating the table name.
2. Names come from the project glossary — the schema is the glossary's most load-bearing consumer.
3. Booleans read as predicates (`is_active`, `has_paid`); avoid negated names (`is_not_expired` invites double negatives).

## Keys & identity

4. Every table has a surrogate primary key, generated by the system (UUIDv7 or equivalent time-sortable ID by default); natural keys get unique constraints, not PK status.
5. Foreign keys are real constraints with explicit `ON DELETE` behavior chosen per relationship — never "the app will keep it consistent".
6. External identifiers (from third-party systems) are stored verbatim in their own column, unique-indexed, never used as the PK.

## Timestamps & lifecycle

7. Every table carries `created_at` and `updated_at` (UTC, set by the database or data layer — one mechanism, used everywhere).
8. **Soft delete by explicit decision only**: tables where history or recovery matters get `deleted_at` (null = live), and every read path must filter it. Tables without that need use hard deletes. Mixing the two styles silently is the bug factory — record the choice per table in the schema comments.
9. Money is integer minor units + currency code; never floats. Time is UTC in storage, localized at the edge.

## Migration-safety rules

10. Every schema change ships as a generated, reviewed, forward-only migration file (`06-delivery/migration-discipline.md` — including the production `db push` ban).
11. **Expand → migrate → contract** for anything in use: add the new shape, backfill/dual-write, then remove the old shape in a later release. Renames and type changes are never done in one step against live data.
12. New columns on existing tables are nullable or defaulted — a required column with no default breaks every in-flight writer.
13. Destructive migrations (dropping columns/tables) state in the PR what proves the data is unused, and run only after a backup is verified restorable (`07-operations/backup-dr.md`).

## Indexing & query performance

14. **Index the columns you filter, sort, and join on — driven by the access patterns, not guessed.** The index set is a function of the queries the app actually runs: the `WHERE` predicates, the `ORDER BY` keys, and the join columns of the real read paths. The list/collection endpoints declare which columns they filter and sort on (`03-design/api-contract-design.md` — the contract's filter/sort surface); those columns are exactly what gets indexed. An index added "because it might help" with no query behind it is write cost for no read.
15. **Composite-index column order is equality columns first, then range and sort columns (left-prefix rule).** A multi-column index serves a query only when the leading columns match the query's equality predicates; a range or sort on a later column can ride the same index, but a column after a range no longer narrows the scan. Order the columns most-selective-equality → range → sort, and remember a composite on `(a, b)` already covers queries on `a` alone — so a separate single-column index on `a` is redundant.
16. **Add a covering index for a hot read path when the win is measured.** When a frequent query reads only a few columns, an index that carries those columns (the key columns plus an include payload) lets the engine answer from the index alone — an index-only scan, no table fetch. Reserve this for paths the latency budget actually cares about (`_spines/performance.md` — the per-route latency target); a covering index is wider and costlier to maintain, so it earns its place with a profile, not a guess.
17. **Every index is a write tax — don't over-index.** Each index must be updated on every insert/update/delete to the indexed columns and consumes storage and cache. The right number is the smallest set that serves the real query mix: prefer one well-ordered composite over three overlapping singles, drop indexes no query uses, and treat "add an index" as a change with a cost side, not a free speedup.
18. **Design the access path to avoid N+1; load in batches, not in loops.** A query that runs once per row of a parent result (the N+1 pattern) is the most common scaling failure: invisible over ten rows, fatal over a million. The fix lives in how the data is fetched — a single join, a batched `IN (...)` / keyset load, or an eager/dataloader strategy that collapses the children into one round trip. This is a design rule because the access pattern, not just the schema, determines whether the model scales (`_spines/performance.md`).
19. **Validate the plan with the query planner against a realistically-sized local dataset.** Before a new index or a hot query ships, read its execution plan (the database's `EXPLAIN` / analyze output) and confirm it uses the intended index and the cost is bounded — a sequential scan or a runaway row estimate is a finding, not a detail. Run this against seed data with production-like cardinality and skew (`04-build/developer-experience.md` rule 9 — the realism target is the plan and the cost, not just the schema); a plan validated over ten rows proves nothing.
20. **Denormalize only with a measured reason, and only over an index that didn't suffice.** Default to normalized (rule, Standards basis); duplicate a fact only when a profiled query against realistic data is still too slow *after* the correct index, and record the trade in the schema (which queries it speeds, which writes now keep the copies consistent). A denormalized column without a measured justification and a stated consistency mechanism is a correctness liability, not an optimization.
21. **Indexes follow one naming convention** so a reader and an agent can tell purpose from name: `ix_<table>_<columns>` for a plain index, `uq_<table>_<columns>` for a unique index, columns in index order, joined by underscores. The name states the table, the kind, and the column order — never an opaque auto-generated string.

## Standards basis

- **Relational normalization** (E. F. Codd; 3NF/BCNF): non-key attributes depend on the key, the whole key, and nothing but the key — transitive dependencies eliminated. Default to 3NF; denormalize only as a measured, recorded decision. Grounds the "each fact stored once" intent behind keys/identity and the one-mechanism timestamp rule.
- **Evolutionary database design & expand→contract** (Ambler & Sadalage, *Refactoring Databases*): structural change as small, reversible, semantics-preserving database refactorings with a transition window of parallel old/new shapes. This is the direct basis of rules 10–12 (forward-only migrations, expand→migrate→contract, additive/nullable columns).
- **UUIDv7 / time-ordered UUIDs** (RFC 9562, which obsoletes RFC 4122): a 48-bit Unix-ms timestamp prefix giving index-friendly, monotonic surrogate keys. Grounds rule 4's "UUIDv7 or equivalent time-sortable ID" default over random v4.
- **PostgreSQL identifier conventions**: lower snake_case unquoted identifiers (avoids case-folding surprises), plural table names by convention. Grounds the naming rules (1, 3).
- **ISO 4217** (currency codes) + **ISO 8601 / UTC** (instants): money as integer minor units plus a currency code, time stored as UTC. Grounds rule 9.
- **B-tree indexing** (Bayer & McCreight, 1972; the default index structure of relational engines): a balanced, sorted tree giving logarithmic point lookups and ordered range/sort scans. Why an index serves equality, range, and `ORDER BY` from one structure, and why the left-prefix rule holds — grounds rules 14–16.
- **"Use the Index, Luke!"** (Markus Winand, use-the-index-luke.com; the SQL-performance material from *SQL Performance Explained*): index design driven by the query's `WHERE`/`ORDER BY`/join columns, composite-column ordering (equality-then-range, the left-prefix rule), covering/index-only scans, and the write cost of each index. Direct basis for rules 14–17 and the EXPLAIN discipline of rule 19.
- **PostgreSQL query planner / `EXPLAIN` documentation** (postgresql.org/docs — "Using EXPLAIN" and "Planner / Optimizer"): reading an execution plan to confirm index use, scan type, and row-estimate accuracy against real data; the canonical form of the planner-validation rule 19. (The cost-based planner and `EXPLAIN (ANALYZE)` interface are the vendor-neutral pattern; every mature relational engine exposes the equivalent.)
- **Relational index design / physical database design** (the access-path-selection literature originating with the System R optimizer, Selinger et al., 1979): indexes and denormalization are physical-design choices justified by the workload's query mix and measured cost, not by schema aesthetics — the basis for the access-pattern-driven framing of rules 14, 18, and the "measure before you denormalize" discipline of rule 20.

## Enforcement
- Mechanism: CI job
- Config: stacks/nextjs-default/ci/pr.yml (migrations job: blocks edits/deletes of applied migration files, forcing the forward-only discipline)
- Fallback if unenforceable: n/a — migration immutability is CI-enforced; naming/keys/timestamp conventions are reviewable in the generated migration diff the same gate forces into the PR.
