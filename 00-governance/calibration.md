# Calibration Register

Every tunable knob in the suite — thresholds, budgets, cadences, score floors, version pins — with its current value, the rationale a future maintainer needs ("why this number"), and a **recalibration trigger**: the observed data that justifies changing it, and in which direction. Seeded by a five-area calibration panel (June 2026) that reviewed all 72 inventoried values: 56 affirmed, 1 adjusted (Lighthouse gate set to the mobile form factor — see the Observations log and CAL-C09 for the why; git history for the how).

**The discipline (root `CLAUDE.md` §5):** changing any knob means updating its row here — and its entry in the machine manifest at the bottom — **in the same commit**. `scripts/check-calibration.sh` extracts the manifest and verifies the register and the tree agree; suite CI runs it on every push/PR, so knob drift is a build failure, not an honor system. The manifest covers every machine-extractable knob including **both sides of each doc↔config pair** (the doc's stated value and the config's enforced value must each match the register).

**Recalibration is data-driven, not mood-driven:** each trigger names what to observe (PR-size distribution, coverage trend, budget burns, sweep findings) — the twice-yearly currency pass (`standards-lifecycle.md` §6.3) walks these triggers against real project data.

Knobs without a manifest entry are register-governed only — their rows above still carry the rationale and trigger, and the currency pass reviews them by hand. What is left unmanifested is genuinely unautomatable: pure-prose cadences ("quarterly", "twice a year", "same day"), per-project example bounds in the operations docs and templates (SLO/SLI targets, RPO/RTO, burn-alert windows — set from each project's measured reality, not a fixed library value), and choices expressed in prose rather than a config literal. Any knob that is a stable literal in a config file is manifested above, even when it appears more than once: the multi-valued artifact-retention and job-timeout knobs are anchored to their unique job/upload-step names rather than left out.
