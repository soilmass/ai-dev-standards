# Developer Experience

The inner loop — clone, run, change, see the result — is the multiplier on every other standard: a slow or fragile setup tax is paid on every task, and for a solo+AI workflow the agent pays it too. This doc makes the environment reproducible and the loop fast, so "works on my machine" is a guarantee, not a hope.

## Reproducible environment

1. **One command to a running app.** A fresh clone reaches a running dev server in a single documented command path (the bootstrap output's next-steps), not a scavenger hunt — `install → bring up local services → migrate → seed → dev`. Every manual prerequisite is either scripted or removed.
2. **The environment is pinned and containerized, not described.** Local services run from a committed `docker-compose.yml` with **pinned images** (calibration CAL-D14), and the editor/runtime is captured in a committed `.devcontainer/devcontainer.json` pinned to the same Node line as CI (CAL-D01). A README paragraph telling you to "install Postgres 17" drifts; a pinned compose file doesn't.
3. **Dev/prod parity.** Local backing services match production in kind and major version (Twelve-Factor X) — the same database engine and major, so a query that works locally works in prod. Production still uses the managed service (`06-delivery/deployment-strategy.md`); local just mirrors its shape.

## A fast, trustworthy inner loop

4. **The loop is fast and incremental.** Hot-reload for the app, watch-mode + the unit tier for tests; the per-change feedback a developer (or agent) waits on is seconds, not a full build. Slow feedback is a defect to fix, like a flaky test.
5. **Seed data is a committed, idempotent script, never a manual ritual.** The `db:seed` script populates a representative-but-minimal dataset (enough to exercise the core loop), is safe to re-run (upsert/conflict-do-nothing), and **never targets production** — it writes rows. A shared SQL dump passed around in chat is not seed data.
6. **The feedback loop is the metric.** DORA/SPACE research ties developer throughput to fast feedback and low friction; treat onboarding-to-running time and inner-loop latency as numbers to keep low, and a regression in either as worth fixing.

## Inner-loop latency as a tracked number

7. **Inner-loop latency has two named components, both measured.** Track *hot-reload time* (save → the change visible in the running app) and *test-feedback time* (save → the relevant unit tier reports pass/fail), not a single vague "feels slow" sense. These are the two waits paid on every edit; you can't keep low what you don't name and observe. A reload that crosses from sub-second into multi-second, or a watch run that drifts past the loop budget, is a measured regression, not a vibe.
8. **A slow inner loop feeds straight into the delivery metrics.** Fast inner-loop feedback is the upstream of the four keys: it shortens lead time (less wall-clock per change) and lowers the change-failure rate (defects caught at the keystroke, not in review or prod). A degraded loop doesn't stay local — it surfaces downstream as slower delivery and more escaped defects, which is why rule 7's two numbers belong in the same conversation as deployment frequency and MTTR.

## Local realism, not just local presence

9. **A local backing service that runs is not a local backing service that's honest.** Matching prod in kind and major version (rule 3) keeps queries *valid*, but a near-empty local store hides the costs that only appear at scale: a missing index, an unbounded query, an N+1 that's invisible over ten rows and fatal over a million. The seed dataset (rule 5) must therefore include enough volume and shape — realistic cardinality, representative skew, the indexes prod actually has — that a query plan locally resembles the plan in production. The realism target is the *plan and the cost*, not just the schema.
10. **The agent pays a compounding feedback tax, and it compounds the worst.** Every probe an AI agent makes — run the test, reload to confirm, re-run after a tweak — blocks on the same loop a human waits on, but the agent issues far more of them per task and cannot context-switch away from the wait. A loop that's a mild annoyance to a developer who tabs to something else is a multiplier on agent task time and token cost, because each slow cycle is dead wall-clock inside the task. Optimizing the inner loop is therefore not a developer comfort item; it is the single highest-leverage way to make agent-driven work cheaper and faster, and a loop regression should be triaged with that multiplier in mind.

## Standards basis

- **The Twelve-Factor App** (12factor.net) — **X. Dev/prod parity** (keep development, staging, and production as similar as possible, same backing-service type and version) is the basis for rules 2–3; **III. Config in the environment** ties the local `DATABASE_URL` to the same boot-validated schema as prod (`04-build/secrets-config.md`).
- **Development Containers specification** (containers.dev) — a committed, pinned, reproducible dev environment as code; the basis for rule 2's `devcontainer.json`.
- **DORA / SPACE / DX Core 4** (dora.dev; the SPACE framework, Forsgren et al.; DX Core 4) — fast feedback loops and low friction are measured drivers of delivery performance and developer effectiveness; grounds rules 4, 6, and 7 (treat inner-loop latency and onboarding time as named metrics). **DORA's four keys** — deployment frequency, lead time for changes, change-failure rate, and time to restore service — are the delivery outcomes; rule 8 traces the inner loop upstream of them: fast keystroke-level feedback compresses lead time and catches defects before they count against change-failure rate. **SPACE** is the reminder that throughput alone is the wrong frame — its five dimensions (Satisfaction, Performance, Activity, Communication, Efficiency-and-flow) put inner-loop friction under E (flow without interruption), the same friction rule 10 says the agent pays in multiplied form. **DX Core 4** (getdx.com; Noda, Forsgren, Storey & Tacho, 2024) unifies DORA, SPACE, and DevEx into four dimensions (Speed, Effectiveness, Quality, Business Impact), carrying the DevEx framework's Feedback Loops emphasis into a measured program — the basis for tracking feedback latency as a first-class number rather than inferring it from output.
- **The Twelve-Factor App — X. Dev/prod parity** (12factor.net), restated for rule 9: parity is not only about the *type and version* of the backing service but about keeping the gap between development and production small across time, personnel, and tools — a local store whose data shape and volume don't resemble prod is a parity gap that hides scale-dependent defects (missing index, N+1) until they reach production.
- **AWS Well-Architected Framework — Operational Excellence pillar** (aws.amazon.com/architecture/well-architected) — its design principle "perform operations as code" and its emphasis on small, frequent, reversible changes with fast feedback are the cloud-architecture framing of the same loop: the faster and more faithful the local cycle, the smaller and safer each change to the running system. Operational Excellence treats the ability to evolve operations through frequent feedback as a pillar property, which is the system-level corollary of rules 7–8 at the team and inner-loop level.
- Pairs with `04-build/testing-strategy.md` (the fast unit tier is the inner-loop test surface and the source of rule 7's test-feedback time) and `06-delivery/deployment-strategy.md` (prod uses the managed equivalent of the local compose services).

## Enforcement
- Mechanism: none-possible
- Config: stacks/nextjs-default/project-config/.devcontainer/devcontainer.example.json + stacks/nextjs-default/project-config/docker-compose.example.yml + stacks/nextjs-default/project-config/db/seed.example.ts (and the nextjs-container equivalents: project-config/docker-compose.example.yml + project-config/prisma/seed.example.ts)
- Fallback if unenforceable: If a change affects local setup or onboarding, the one-command bootstrap still works — the pinned devcontainer/compose and the idempotent seed script are updated in the same change, and dev still mirrors prod's backing-service major.

## Bootstrap
- What new-project.sh injects for this standard: `.devcontainer/devcontainer.json`, `docker-compose.yml` (pinned local Postgres + adminer), and the idempotent `db/seed.ts` (Prisma: `prisma/seed.ts`) with a `db:seed` script — a fresh clone is one `docker compose up -d` + install + migrate + seed from a running app.
