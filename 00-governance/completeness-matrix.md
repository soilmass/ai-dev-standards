# Completeness Matrix

The live grid: every layer cell of the suite against the four-test quality bar (see `README.md`). Each cell links the satisfying file or names the gap explicitly — a cell is never silently blank. The executable subset of this matrix is `scripts/audit-completeness.sh` (footer presence + fallback aggregation); **re-run it after adding or changing any standard, and update this grid in the same commit.**

Column meanings — **Articulated**: rule written concretely. **Templated**: fill-in artifact exists where one is the deliverable. **Enforced**: mechanical gate, or declared `none-possible` with its fallback aggregated into the self-review checklist. **Bootstrappable**: `new-project.sh` injects it, or the doc states "reference only".

## Known gaps (explicit, not blank)

1. **Lifecycle & pinned-decisions currency** have no mechanical gate — they depend on the twice-yearly currency pass (`00-governance/standards-lifecycle.md` §6.3). Accepted: a cron can't read the ecosystem.
2. **Branch protection, tag protection & signing** are host-side settings `new-project.sh` cannot set during bootstrap; they are now two injected commands away — `scripts/setup-branch-protection.sh` applies branch protection (required checks derived from `pr.yml`), required signed commits, and a `refs/tags/v*` protection ruleset via the GitHub API, and `scripts/configure-signing.sh` turns on commit/tag signing — but both still have to be run once, with `gh` authenticated, after the first push (`_spines/version-control.md`). The manual fallback checklist is in `05-verification/ci-pipeline.md` § "Branch protection — the one manual step".
3. **SLO/RUM tooling** is per-project (needs real traffic and a chosen vendor); the standards define shape and triggers only.

4. **Chaos engineering / game days** are not standardized; the solo-scale equivalent — the pre-launch full-loss DR drill (`07-operations/backup-dr.md`) and the quarterly rollback drill (`06-delivery/rollback.md`) plus design-time failure-mode analysis (`_spines/reliability.md`) — is in place. Full fault-injection is team-scale; revisit if/when traffic and team justify it.
5. **A/B testing / experimentation** (variant assignment, guardrail metrics, statistical significance) is absent — a post-traffic concern. Feature flags (`06-delivery/release-process.md`) cover dark-ship/rollout; the experimentation framework is deferred until there is traffic to experiment on.
6. **AI/LLM feature engineering** is a deliberate **planned future spine**, not yet built — the library's current scope is *developing with* AI coding agents, not *building* AI-powered product features. A project shipping an LLM feature today is already covered by the existing ADR (`01-context/adr.template.md`), threat-model (prompt injection, `03-design/threat-modeling.md`), rate-limiting/cost (`03-design/rate-limiting-abuse.md`, `_spines/cost.md`), and observability (token metrics, `07-operations/observability.md`) standards; the AI-specific concerns (prompt versioning, eval gates, RAG architecture, agent safety, model lifecycle) await the spine when scope expands. Recorded here so the gap is explicit, not silent.

When adding a standard: write the doc with its footer → wire/point enforcement → add its row here → run `scripts/audit-completeness.sh`.
