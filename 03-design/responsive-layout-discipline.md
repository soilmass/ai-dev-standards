# Responsive Layout Discipline

Layout, spacing, and typographic micro-detail are where a competent build separates from a polished one — and where regressions hide, because a page that drifts off its grid or collapses at one breakpoint still passes every functional assertion. This doc owns the structural rules for *how* a layout is composed and *at how many viewports it must hold*. The concrete breakpoint widths, spacing steps, and tolerance are a stack **choice** living in the preset's Playwright config (`stacks/nextjs-default/project-config/playwright.config.example.ts`); the rules below are the **standard** and hold regardless of those values. The token/no-bare-hex discipline and the component hierarchy are owned by `03-design/ui-design-system.md`; the accessibility floor (target size, focus, contrast) by `03-design/ui-accessibility-patterns.md` and `05-verification/a11y-perf-gates.md`; this doc does not restate them.

## Mobile-first breakpoint discipline

1. **Layout is authored mobile-first.** The base, unqualified styles target the smallest supported viewport; every breakpoint is a `min-width` query that *adds* capability as space grows. Authoring desktop-first (`max-width` queries peeling capability away) is a violation — it inverts the cascade, makes the small-screen case the exception rather than the default, and is the single most common cause of a layout that "works on my monitor" and breaks on a phone.
2. **Breakpoints are declared in one ascending order and used in that order.** The preset names an ordered set of `min-width` breakpoints (smallest → largest); a component opts into a wider layout by adding the next breakpoint up, never by hand-writing a one-off pixel query. Two components reaching the same visual change at two different ad-hoc widths is the drift this rule forbids — the breakpoint set is the shared vocabulary, and a new breakpoint is a deliberate addition to that set, not an inline magic number.
3. **Breakpoints follow content, not devices.** A breakpoint is added where the *layout itself* breaks — a line wraps badly, a grid gets too sparse or too dense — not to chase a specific phone or tablet model. Device dimensions churn; content reflow needs are stable. The named set is sized to cover the content's real reflow points, and the visual baselines (rule 9) are shot at widths that span them.

## Spacing and grid scale

4. **Spacing comes from one documented scale — never arbitrary pixel values.** Margins, padding, and gaps are drawn from a single spacing scale defined once in the styling system (a geometric ramp or a fixed baseline grid, e.g. multiples of a 4px base unit). A raw `margin: 13px` or `gap: 22px` that does not land on a scale step is a violation, the same way a raw hex color is (`03-design/ui-design-system.md` rule 2). Consistent spacing is what makes unrelated components feel like one product.
5. **Layout structure uses the grid/flex system, not absolute positioning or magic offsets.** Multi-column and aligned layouts are expressed through the styling system's grid/flexbox primitives with scale-derived gaps; absolute positioning and negative-margin nudges are reserved for genuine overlays and are never load-bearing for the main reading flow. A layout held together by hardcoded offsets breaks the moment content length or viewport width changes.
6. **Vertical rhythm is intentional, not incidental.** The vertical spacing between stacked blocks comes from the same scale and is applied in one direction (a consistent block-flow margin convention), so adjacent components do not double-space or collapse unpredictably. Rhythm is part of the scale, not a per-component afterthought.

## Measure, line-height, and typographic micro-detail

7. **Reading content has a bounded measure.** Long-form text columns are capped at a comfortable line length (the typographic "measure", conventionally ~45–75 characters) via a max-width on the text container — never allowed to span the full width of a wide viewport, where lines grow too long to track from end to start. The cap is expressed in character-relative units where the styling system supports it, so it scales with the type size.
8. **Line-height and letter-spacing are set deliberately, scaled to role.** Body copy carries a generous unitless line-height (room to read); headings are tighter; letter-spacing is adjusted only where the type size or weight calls for it (slightly looser for small caps / all-caps, tighter for large display sizes). These are set once per type role in the scale, not sprinkled per component. Default-browser line-height on long body copy, or all-caps with no added tracking, is the micro-detail miss this rule catches.
9. **Layout is verified at multiple viewports, never one.** A surface is correct only if it holds across the breakpoint set — at minimum a small-phone width, a tablet width, and a desktop width. The visual-regression suite shoots each guarded surface at every breakpoint width in the preset's viewport set (one Playwright `visual-*` project per width), so a layout that holds at desktop but collapses at the phone width fails its baseline at that width. Reviewing or signing off a layout at a single viewport is the gap this rule closes: "looks right" means "looks right at every width it ships to".

## Standards basis

- **Mobile-first / progressive-enhancement layout** (Luke Wroblewski, *Mobile First*; the responsive-web-design lineage from Ethan Marcotte): author the constrained case first and add capability with `min-width` queries as space grows. Grounds rules 1–3 — base styles target the smallest viewport, breakpoints are ascending `min-width` additions placed where content reflows, not where a device sits.
- **Modular / baseline spacing scales** (the 4px/8px baseline-grid convention widely codified in modern design systems; the broader typographic baseline-grid tradition): spacing as a small set of scale steps rather than arbitrary values is what produces visual consistency. Grounds rules 4–6 — one spacing scale, grid/flex structure over magic offsets, intentional vertical rhythm.
- **Typographic measure and rhythm** (Robert Bringhurst, *The Elements of Typographic Style*: the ~45–75-character measure, line-height proportioned to measure and size, restraint with letter-spacing). Grounds rules 7–8 — bounded measure on reading columns, line-height/letter-spacing set per type role.
- **Responsive verification across viewports** (the visual-regression discipline in `05-verification/visual-regression.md` specialized to width): a layout claim is only as good as the widths it was checked at, so the same surface is captured at several breakpoint widths and each is a committed baseline. Grounds rule 9 — multi-viewport capture as the mechanical backstop.

## Enforcement
- Mechanism: none-possible
- Config: stacks/nextjs-default/project-config/playwright.config.example.ts (the per-viewport `visual-*` projects — one per breakpoint width, the CAL-C15 viewport set — partially back rule 9 by failing a baseline when a layout drifts at a given width; run by stacks/nextjs-default/ci/nightly.yml). The mobile-first order, scale adherence, measure, and line-height/letter-spacing discipline (rules 1–8) have no mechanical gate — a pixel diff catches drift, not authoring-rule violations.
- Fallback if unenforceable: Confirm the layout is authored mobile-first (base styles target the smallest viewport; breakpoints are ascending min-width additions placed at content-reflow points), all spacing/gaps come from the documented scale (no arbitrary pixel values, structure via grid/flex not magic offsets), reading columns are capped to a ~45–75-character measure, line-height/letter-spacing are set per type role, and the surface has been verified — by visual baseline or by hand — at a small-phone, a tablet, and a desktop width, not just one.

## Bootstrap
- What new-project.sh injects for this standard: the per-viewport `visual-*` Playwright projects (the breakpoint-width viewport set) via the injected `playwright.config.ts`, and the example multi-viewport visual spec under `tests/visual/`; the nightly workflow runs every `visual-*` project. The authoring rules themselves are reference-only — they ride the self-review checklist (`05-verification/code-review-standard.md`).
