# Graphics & Asset Cohesion

Every image, icon, illustration, and social-card asset a site references must **resolve, render in the right format at the right size, and belong to one coherent visual language**. This doc owns the asset layer that sits between the design system (`03-design/ui-design-system.md`, which owns tokens and component structure) and the accessibility baseline (`03-design/ui-accessibility-patterns.md` + `05-verification/a11y-perf-gates.md`, which own behavior and the automated a11y floor). It splits into rules a machine can decide (the asset resolves; the format and `sizes` are correct) and rules only a human or LLM judge can decide (the assets cohere; the alt text is *meaningful*, not merely present).

The motivating defect: an Open Graph image referenced as `<meta property="og:image" content="/og/home.png">` while no `/og/home.png` exists on the built site — a 404 that no per-page render check catches because the markup is well-formed. Presence of the tag is not presence of the asset.

## Mechanically-checkable rules (the asset layer's gate)

1. **Every referenced asset resolves on the built site — zero 404s.** Each `src`, `srcset` candidate, `<source srcset>`, CSS `url(...)`, `<link rel="icon">`/`apple-touch-icon`, web-app-manifest `icons[].src`, and every `og:image` / `twitter:image` / `og:image:secure_url` URL must return `2xx` for a real asset of the declared type — not an HTML error page served with `200`. This is the same broken-asset class the multi-page site-integrity gate enforces for links (`03-design/multi-page-site-coherence.md`); asset URLs are crawled alongside internal links, and the allowed-asset-404 count is **0** (the same floor as link 404s — see that doc's Enforcement). A referenced-but-absent OG image, a typo'd icon path, or a CDN asset removed out from under a hardcoded URL all fail here.
2. **Modern format with a correct fallback.** Raster photographic/illustrative content is served as **AVIF or WebP with a fallback** the browser can use (a `<picture>` with `<source type="image/avif">` → `type="image/webp">` → `<img>` PNG/JPEG, or the framework image component that emits this automatically). **Icons and flat-color line art are SVG** (or an icon font), never a rasterized PNG sprite, so they stay crisp at every density and inherit `currentColor`. A bare `<img src="hero.png">` for a large photographic hero, or a PNG used where an SVG icon belongs, is a violation. The declared MIME type must match the bytes (a `.png` that is actually a JPEG fails rule 1's type check).
3. **Responsive `sizes` and intrinsic dimensions are present and correct.** Any image that is not a fixed-size icon declares `width`/`height` (or an aspect-ratio box) so it reserves layout space and does not shift content (the CLS budget in `05-verification/a11y-perf-gates.md`). A responsive image with `srcset` width descriptors (`w`) **must** carry a `sizes` attribute that reflects the image's *rendered* width at each breakpoint — a `sizes` that always says `100vw` for an image rendered at one-third width over-fetches on every viewport and is a defect, not a nicety. Fixed-layout images use density descriptors (`1x`/`2x`) or `sizes` may be omitted.
4. **`alt` is present on every content image; decorative images have empty `alt`.** Every `<img>` and SVG-as-image either carries an `alt` attribute (content) or `alt=""` plus `role="presentation"` / `aria-hidden` (decorative). A missing `alt` attribute is a violation a linter and the axe a11y gate (`05-verification/a11y-perf-gates.md`) both catch. *Whether the alt text is meaningful* is rule 7 below — not mechanically decidable.

## Judgment rules (none-possible — human/LLM-judge review)

5. **OG / social-card images cohere across pages.** Every page's `og:image` is a real, intentional card — correct dimensions (the 1200×630 convention, ≥ 1200px wide), legible at thumbnail size, and visually consistent with its siblings (shared template, type treatment, logo placement, and palette drawn from the design tokens). A homepage with a polished branded card while interior pages fall back to a stretched logo or a default framework placeholder fails cohesion even when every URL resolves (rule 1). No machine decides "these cards look like one family."
6. **Icons and illustrations speak one visual language.** All icons come from **one** set / stroke weight / corner radius / fill-vs-stroke convention (or a deliberately documented pairing), and illustrations share one style — not a mismatched grab-bag of a line-icon here, a filled glyph there, and a 3-D render elsewhere. New assets are drawn or sourced to match the established language; an agent adding an icon picks from the in-use set rather than introducing a third style. This is the asset-layer expression of the single-styling-system rule (`03-design/ui-design-system.md` rule 1).
7. **Alt text is meaningful, not merely present.** `alt` describes the image's *purpose in context* — what a sighted user gains from it — not the filename, not "image", not a keyword-stuffed caption. A chart's alt states the takeaway; a decorative flourish is correctly `alt=""`; a logo's alt is the company name, not "logo". Rule 4 proves the attribute exists; only a human (or an LLM judge reading the surrounding content) can decide the text actually conveys the image's meaning.

## Standards basis

- **Open Graph protocol** (ogp.me) — `og:image` and the image sub-properties (`og:image:width`/`:height`/`:secure_url`); grounds rule 1's requirement that the referenced card actually resolve and rule 5's dimension/coherence expectations. **Twitter/X Cards** (`twitter:card`, `twitter:image`) corroborate the 1200×630 large-summary-card convention.
- **MDN — Responsive images & the `<picture>` element** (developer.mozilla.org/en-US/docs/Web/HTML/Guides/Responsive_images): the `srcset` width-descriptor (`w`) + `sizes` contract and the `<picture>`/`<source type>` format-fallback pattern that ground rules 2–3. The `sizes` attribute must describe the rendered width so the browser picks the smallest sufficient candidate.
- **WebP / AVIF** (web.dev — "Serve images in modern formats", "Use AVIF/WebP"): AVIF and WebP deliver the same visual quality at materially smaller byte sizes than JPEG/PNG, with `<picture>` supplying the legacy fallback; grounds rule 2's modern-format-with-fallback requirement and the performance link to the LCP budget.
- **SVG for icons / scalable line art** (W3C SVG 2): vector icons stay sharp at any density, scale without new assets, and inherit `currentColor` for theming; grounds rule 2's "icons are SVG, not raster" rule.
- **WCAG 2.2 — 1.1.1 Non-text Content (Level A)** (w3.org/TR/WCAG22/) and the **W3C/WAI alt-text decision tree** (w3.org/WAI/tutorials/images/decision-tree/): every non-text content item has a text alternative serving the equivalent purpose, and purely decorative images are marked so AT ignores them. Grounds rule 4 (presence/empty-alt) and rule 7 (meaningfulness). The "automated checks catch ~half of WCAG" caveat in `05-verification/a11y-perf-gates.md` is exactly why alt *quality* (rule 7) is none-possible while alt *presence* (rule 4) is lint/axe-gated.
- **Core Web Vitals — CLS** (web.dev): images without reserved dimensions are a primary layout-shift source; grounds rule 3's width/height/aspect-ratio requirement and its tie to the CLS budget in `05-verification/a11y-perf-gates.md`.

## Enforcement
- Mechanism: none-possible
- Config: stacks/nextjs-default/project-config/scripts/check-site-integrity.example.sh (the built-site crawl gate from `03-design/multi-page-site-coherence.md` — it resolves every referenced image/OG/icon/manifest asset URL alongside internal links at the 0-404 floor) + stacks/nextjs-default/lint-config/biome.json (a11y lint: missing-`alt` and image-element rules) + stacks/nextjs-default/ci/lighthouserc.json (CLS budget catches unsized images). The mechanical half (rules 1–4) is gated there; rules 5–7 are not machine-decidable.
- Fallback if unenforceable: Every page's OG/social-card and the site's icons and illustrations form one coherent visual language — OG cards share a template, dimensions, and palette across pages (no stretched-logo or framework-default fallback), and icons/illustrations come from one set/style rather than a mismatched grab-bag.
- Fallback if unenforceable: Every content image's alt text conveys the image's purpose in context (the chart's takeaway, the logo's company name) rather than a filename, "image", or keyword stuffing, and every decorative image is correctly empty-alt — presence of the attribute is lint-checked; its meaning is verified by reading it against the surrounding content.
