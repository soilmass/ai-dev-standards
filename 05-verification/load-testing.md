# Load & Capacity Testing

The Lighthouse gate (`05-verification/a11y-perf-gates.md`) measures one session in a lab; it says nothing about behavior under hundreds of concurrent users. This doc covers performance **under load** — where the system saturates, whether it holds its latency SLO at peak, and how much headroom exists before it falls over. It is a **pre-launch and pre-traffic-event** activity, not a per-PR gate.

## What and when

1. **Test the revenue-critical paths under realistic concurrency** — the same journeys the E2E crown guards (`04-build/testing-strategy.md`), driven at expected peak load, not a single synthetic hammer on one endpoint. The script models real user behavior (think-time, mixed actions), because uniform max-rate traffic finds different limits than real load.
2. **Derive the pass/fail bar from the SLO, not a vanity number.** The test passes if the **latency SLO holds (p95/p99) and the error rate stays in budget** at the target load with headroom (e.g. 2× expected peak); it fails if either breaches. The SLO (`07-operations/slo-error-budgets.md`) is the contract the load test verifies.
3. **Run it at the moments load changes, not continuously:** before first launch, before a known traffic event (campaign, launch, seasonal peak), and after an architecture or capacity-relevant change. Between those, field RUM + SLO burn alerts (`_spines/performance.md`) are the ongoing signal.

## The kinds of test

4. **Use the right shape for the question:** **load** (steady expected peak — does the SLO hold?), **stress** (ramp past peak — where's the breaking point and does it fail gracefully?), **spike** (sudden surge — does it absorb or shed load cleanly?), and **soak** (sustained load for hours — memory/connection leaks, resource creep). A launch needs at least a load + a soak; stress/spike where a surge is plausible.
5. **Capacity planning falls out of the stress test.** Find the **knee** (where latency degrades non-linearly), know your headroom above expected peak, and set autoscaling bounds / connection-pool / rate limits (`03-design/rate-limiting-abuse.md`) from measured numbers — not guesses. "We can handle launch" is a measurement, not a hope.
6. **Test in a production-like environment, and degrade safely if testing prod.** Load against a prod-shaped environment with prod-shaped data; never blind-hammer production. Coordinate with rate limits and have a kill-switch — a load test that takes down the real service is an outage you scheduled.

## Model the load profile from real data

7. **Shape the synthetic load from observed traffic, not a uniform max-rate firehose.** The mix of actions, the ratio of reads to writes, the concurrency distribution, and the think-time between steps come from production RUM and access logs (`07-operations/observability.md`) — or, pre-launch, from the closest analogue and a documented assumption. A flat hammer on one endpoint exercises a load the system will never actually see and misses the contention that real journeys create (a checkout that reads a catalog, holds a cart, then writes an order stresses different resources in sequence than a thousand identical GETs).
8. **Reproduce the real arrival pattern, not just the average rate.** Traffic arrives in bursts, not at a constant interval; derive the peak-to-mean ratio and the burst shape from the log timeline and feed it to the generator (open-model arrival rate, not a fixed pool of looping virtual users). Tail latency is set by the bursts, so a test that smooths them away reports a latency the system never delivers under real conditions.
9. **Match the data and cache state to production.** Run against a dataset of production-like cardinality and distribution, with caches warmed (or cold-started deliberately if testing a cold path) — a query that is instant against a thousand seeded rows can be the knee against ten million. State the seeded volume and cache assumptions in the test record so the result is interpretable later.

## Capacity planning from the stress test

10. **Find the knee, then set headroom from it — don't extrapolate from the linear region.** Ramp the stress test until latency and error rate climb non-linearly; that inflection is the knee, the practical ceiling. Capacity headroom is the gap between expected peak and the knee, and it must absorb the modeled burst peak (rule 8) plus a safety margin for the unmodeled — not just the steady-state mean.
11. **Convert the measured ceiling into the concrete limits other layers consume.** The knee and the per-instance saturation point set autoscaling bounds, the connection-pool ceiling, worker counts, and the rate-limit / load-shed thresholds (`03-design/rate-limiting-abuse.md`) — each derived from a measured number with the measurement cited, never a round-number guess. When the bottleneck is a shared backing resource (database connections, a downstream quota), horizontal scaling of the app tier past that point buys nothing; identify which resource the knee actually belongs to.
12. **Re-derive capacity when the inputs move.** The headroom figure is only valid for the architecture, instance shape, and traffic mix that produced it; a capacity-relevant change (new dependency on a hot path, instance-type change, a traffic mix shift seen in RUM) invalidates the number and triggers a re-run before the next traffic event.

## Test the degradation journey, not just survival

13. **Verify the system sheds load gracefully under overload — push past the knee on purpose.** A stress test that stops at the knee proves where the ceiling is; pushing beyond it proves *how the system behaves there*. The pass condition is graceful degradation: shed or queue excess work, protect the core path, and recover when load subsides — not a cascading failure, not a crash, not unbounded latency growth (`_spines/reliability.md`).
14. **Confirm the circuit breaker opens AND the user lands on a working fallback.** "The breaker tripped" is half the test. Drive the dependency to failure under load and assert the *user-visible* outcome of the journey: a fast, honest fallback (cached result, queued-for-later acknowledgement, a clear degraded-mode message) — not a spinner that hangs to timeout, a 500, or a blank screen. Test the journey through the failure, including recovery when the dependency returns; a breaker that opens but strands the user is a failed degradation, even though the system "survived".
15. **Assert retry and timeout behavior under load, not just in isolation.** Synchronized client retries against a struggling dependency are a load amplifier (the retry storm that turns a blip into an outage). The degradation test confirms timeouts are bounded, retries are jittered/capped, and the aggregate retry load does not itself push the system past the knee.

## Soak: what to watch over hours

16. **Soak at sustained realistic load for hours, long enough for slow leaks to surface** — a few minutes of steady traffic proves throughput, not endurance. The duration must be long enough to expose the slow failure modes that a short test hides; size it to the slowest cycle you care about (a full cache TTL turnover, a scheduled-job interval, a token-refresh cycle).
17. **Watch the resources that creep, not just the latency line.** Over the soak, trend: heap/memory growth and GC pause frequency (the leak that only shows as a slow climb), connection-pool and file-descriptor occupancy (exhaustion that only bites after hours of churn), cache hit-rate and eviction behavior as the working set turns over, and queue depth / backlog drift. A flat p95 with a steadily rising heap is a failing soak, not a passing one — the failure is just scheduled for later.
18. **A soak passes only if the trends are flat or bounded across the full window.** Throughput and latency holding is necessary but not sufficient; a resource curve that has not plateaued by the end of the window is an unbounded growth signal — extend the soak or treat it as a leak to fix before launch, not a result to round off.

## Standards basis

- **Google SRE — "Handling Overload", "Addressing Cascading Failures", and "Non-Abstract Large System Design"** (*Site Reliability Engineering*, O'Reilly): client-side throttling and load shedding (Handling Overload) ground the graceful-degradation pass condition and the retry-storm guard (rules 13, 15); the cascading-failures chapter's overload/retry-amplification and recovery dynamics ground the push-past-the-knee and degradation-journey rules (rules 13–15); NALSD's capacity estimation and iterative-design method grounds deriving concrete limits from measured numbers (rules 10–12). **Google SRE — "Capacity Planning"** (*The Site Reliability Workbook*) adds the demand-forecast-then-provision-with-headroom discipline behind rules 10 and 12. Pairs with `_spines/reliability.md`.
- **The four load-test types — load / stress / spike / soak** (the standard performance-testing taxonomy, e.g. k6 and Gatling test-type guides): rule 4; the stress type is where the knee and the degradation journey are exercised (rules 10, 13), and the soak type is the endurance/leak test (rules 16–18).
- **Little's Law / queueing theory** (L = λW — average concurrency = arrival rate × average time in system) — the formal basis for capacity reasoning: it converts the modeled arrival rate (rule 8) and the target latency into required concurrency, and explains why latency climbs non-linearly toward the knee as utilization approaches 1 (Kingman's formula / the M/M/1 response-time curve). Grounds rules 8, 10, and 11. **Open vs. closed workload models** (the queueing distinction behind rule 8): a closed model of looping virtual users self-limits offered load when the system slows and understates tail latency, whereas an open arrival-rate model preserves the real burst pressure.
- **RUM-derived workload modeling** — the principle that a representative load is built from observed production behavior (action mix, arrival pattern, data cardinality) rather than a uniform synthetic rate; the field truth comes from real-user monitoring and access logs (`07-operations/observability.md`), the basis for rules 7–9 and 12.
- **Core Web Vitals field data** (`_spines/performance.md`, `07-operations/observability.md`) — the lab/load tests approximate; field RUM at p75 is the ongoing truth between load tests and the source the load profile is modeled from.
