# Static Analysis & Image Scanning

The pipeline already scans **secrets** (gitleaks) and **dependencies** (audit + dependency-review + SBOM); this doc adds the missing security layer — **static analysis of your own code** (SAST) and **vulnerability scanning of the artifact you ship** (the container image). Together they close the secure-SDLC controls that a compliance framework (`_spines/compliance.md`) expects.

## Static application security testing (SAST)

1. **Every PR is statically analyzed for security defects, as a required check.** SAST inspects the source for injection, unsafe deserialization, path traversal, weak crypto, SSRF, and similar CWE-classed flaws that lint and type-checks don't catch — and it gates merge like any other required check (`05-verification/ci-pipeline.md`). The engine is a stack choice (CodeQL in the presets; Semgrep is the documented alternative), recorded as an ADR.
2. **Run the security query suite, not the default.** Use the deeper/security-extended rule set so the analysis finds the vulnerability classes, not just code-quality smells; findings surface in the platform's security dashboard with severity and a remediation path.
3. **A finding is triaged, not ignored.** A true positive is fixed; a false positive is dismissed *with a recorded reason* (the same accept-with-justification discipline as a dependency exception, `04-build/dependency-policy.md`) — never silently muted. SAST is noisy by nature, so the triage discipline is what keeps the gate trusted.

## Image & artifact scanning

4. **A published artifact is scanned before it ships.** Any preset that builds a container image scans it for known OS- and library-layer CVEs and **fails the release on fixable HIGH/CRITICAL** vulnerabilities (`ignore-unfixed` so an unpatchable upstream CVE doesn't block forever, but is still visible). A serverless preset has no image to scan — the SAST + dependency layers are its coverage.
5. **Scan the same digest you attest and deploy.** The scan, the SLSA provenance attestation, and the deploy all reference the one built image digest (`04-build/supply-chain.md`) — scanning a different build than you ship proves nothing.
6. **Infrastructure-as-code is scanned for misconfiguration where it exists.** Projects that adopt IaC add config scanning (tfsec/checkov/trivy-config) as a gate (`06-delivery/infrastructure-as-code.md`); the managed/serverless presets ship no IaC, so this is reference-only until then.

## Triage workflow & false-positive discipline at scale

7. **Every finding ends in a verdict, and only two verdicts close it: fixed or dismissed-with-reason.** A finding left in "open" indefinitely is the failure state — it means the gate produced work nobody owns. Triage runs at the point the finding appears (on the PR), not in a quarterly cleanup; the diff that introduces a flaw is the cheapest place to resolve it (`05-verification/code-review-standard.md`).
8. **A true positive is fixed in the same change that introduced it, or it blocks merge.** The verdict "true positive" is reserved for findings where the engine has correctly identified a reachable weakness: tainted external input reaches a sink, the unsafe API is actually called with attacker-controlled data, the weak primitive guards something that matters. A true positive is not downgraded by inconvenience — if it can't be fixed in the diff, it becomes a recorded, expiring exception (rule 9), never a silent mute.
9. **A false positive is dismissed with a written reason, recorded where the next reader will see it.** A dismissal reason names *why* the finding doesn't hold — the sink is not reachable from untrusted input, the value is a compile-time constant, the path is dead, the rule misclassified the framework's safe-by-construction API. The reason is attached to the dismissal in the analysis platform (or an inline suppression annotated with the justification), carrying the same accept-with-justification weight as a dependency exception (`04-build/dependency-policy.md`). An unexplained dismissal is indistinguishable from hiding a real bug, so it is not permitted.
10. **Suppressions are narrow and attributed, never blanket.** A suppression targets the specific finding (rule + location), not a whole rule, file, or directory; a disabled rule is a config decision recorded as an ADR (`04-build/coding-standards.md`), not a per-diff convenience. Broadening a suppression's scope to silence noise is how a gate quietly stops finding anything.
11. **When finding volume erodes trust, tune the ruleset — don't lower the bar.** A gate that floods every PR with low-value findings trains reviewers to dismiss without reading, which is worse than no gate: the one true positive drowns. The response is to tune precision (raise the severity floor that blocks merge, scope or retire the chronically-wrong rule via ADR, baseline pre-existing findings so the gate measures *new* defects) — not to make dismissal a reflex. The recorded dismissal rate is the health signal: a sustained high false-positive rate is a ruleset bug to fix, tracked as library flow-back (`00-governance/flow-back-log.md`) when it recurs across projects.

## Research vs. gate: where strictness lives

12. **Local analysis is permissive; the CI gate is strict.** A developer running the analyzer on their machine should get fast, broad, exploratory feedback — including low-severity and noisy classes — so problems surface while the code is still warm. The merge gate runs the agreed ruleset at the agreed severity floor and fails deterministically on it. Conflating the two (gating on every local-research finding, or relaxing CI to match a lenient local run) breaks both: the gate becomes flaky or toothless, and exploration gets punished.
13. **The gate's ruleset is pinned and reproducible; research mode may float.** What blocks merge is a named, version-pinned query suite at a recorded severity floor, so the same code yields the same verdict on any machine and any day (`05-verification/ci-pipeline.md`). Engineers may run newer or experimental rules locally to look ahead, but a finding only *gates* once it's promoted into the pinned CI suite — adding a rule to the gate is a deliberate change, not a silent upgrade that turns a green main red overnight.
14. **Confirm the gate is actually running the security suite, not silently degraded.** A SAST step that runs the default/quality ruleset, skips on a misconfigured matrix, or passes because the security pack failed to load is a gate that proves nothing. The pipeline asserts the security query suite is the one that ran — fail the job if the expected suite is absent (rule 2) — so "green" cannot mean "scanned nothing."

## Standards basis

- **OWASP — SAST & the Top 10** (owasp.org; the Source-Code-Analysis-Tools page) — static analysis as the developer-time control for the injection/access/crypto vulnerability classes; the basis for rules 1–3. Findings map to **CWE** (cwe.mitre.org), the common weakness taxonomy.
- **CodeQL** (codeql.github.com) and **Semgrep** (semgrep.dev) — the query-based SAST engines behind rule 1 (CodeQL in the presets, named as the canonical choice; the engine is a per-project ADR).
- **Trivy** (aquasecurity.github.io/trivy) and **container image scanning** — OS/library CVE scanning of the built image; the basis for rules 4–5. **CIS Benchmarks** and **tfsec/checkov** ground rule 6's IaC misconfiguration scanning.
- **OpenSSF Scorecard** (`SAST`, `Vulnerabilities`, `Fuzzing` checks) and **NIST SSDF (SP 800-218) — PW.7 (review/analyze human-readable code) and PW.8 (test executable code)** — the secure-development practices this doc satisfies: PW.7 is the SAST gate (rules 1–3, 7–14), and the review-and-test pairing is why findings are triaged to a verdict rather than merely produced. Complements `_spines/security-privacy.md` and the dependency/supply-chain layer (this doc is *code + image* scanning, those are *secrets + dependencies*).
- **OWASP SAMM v2** (owaspsamm.org) — the **Verification → Security Testing** practice frames automated SAST as a maturity activity whose value comes from *acting on results*, not from running the tool: triage to a verdict (rules 7–9) and tuning to keep signal high (rule 11) are the SAMM behaviors that move the practice up a maturity level. The research-vs-gate split (rules 12–14) is the SAMM distinction between deep manual/exploratory review and the automated baseline that gates every change.
- **OWASP ASVS 5.0** (asvs.dev) — the verification-requirements standard; Chapter **V15 (Secure Coding and Architecture)** sets the secure-coding requirements (defensive-programming patterns, safe handling of untrusted input) that static analysis verifies in code. ASVS specifies *what to verify*, not *which tool* — it doesn't mandate SAST — so the security query suite (rule 2) is the mechanism that checks code against its per-requirement structure, and the CWE mapping below is how a finding traces to a verifiable requirement rather than a vibe.
- **CWE** (cwe.mitre.org) — the common-weakness taxonomy every SAST finding is classed against (injection CWE-89/CWE-78, path traversal CWE-22, unsafe deserialization CWE-502, SSRF CWE-918, weak crypto CWE-327). The true/false-positive verdict (rules 8–9) is a judgment about whether the named CWE is actually reachable in this code; **CWE Top 25 Most Dangerous Software Weaknesses** is a defensible basis for the severity floor that blocks merge (rule 11).

## Enforcement
- Mechanism: CI job
- Config: stacks/nextjs-default/ci/pr.yml (codeql job — SAST, both presets) + stacks/nextjs-container/ci/release.yml (Trivy image scan on the published image)
- Fallback if unenforceable: n/a — SAST is a required PR check and image scanning gates the release; triage of findings rides the dependency-exception discipline in the self-review checklist.

## Bootstrap
- What new-project.sh injects for this standard: the CodeQL SAST job (in the injected `pr.yml`, a required check once branch protection is applied) and, for the container preset, the Trivy image scan in `release.yml`.
