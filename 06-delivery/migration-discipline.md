# Migration Discipline

Database changes are the least reversible thing a web project does routinely. These rules make them boring.

## The rules

1. **Forward-only.** Every schema change is a new migration file. There is no "down in production" — recovery from a bad migration is a *new* forward migration (or a restore, `07-operations/backup-dr.md`), never running a down script against real data.
2. **Never edit an applied migration.** Once a migration has run anywhere shared (preview DB, production), the file is immutable history. Fixing a mistake means a new migration. CI blocks modification or deletion of existing migration files.
3. **Generated, then reviewed.** Migrations are generated by the ORM's tooling from the schema source, then **read by a human/agent before merge**: check lock behavior, data loss, and that the SQL says what the schema diff means. Generated ≠ trusted.
4. **NEVER run `db push` (schema sync) in production.** Both major ORMs expose a push/sync mode for prototyping only — it diffs live and applies immediately, bypassing files, review, and history. Production changes go through generated, reviewed migration files, applied by the deploy pipeline. Push is for throwaway local databases, nothing else.
5. **Dry-run before apply.** Migrations run against a production-shaped database (preview/scratch restored from backup) before they run in production; the pipeline applies them in order, exactly once, recorded in the migrations table.
6. **Migrations ship with their code** in the same PR, ordered so the migration is compatible with both the old and new code (expand → migrate → contract, `03-design/data-modeling.md` rules 11–13).
7. **Destructive operations** (drop column/table, type narrowing) additionally require: evidence of non-use in the PR, a verified-restorable backup, and a contract-phase release separate from the expand phase.

## Agent rules

- Schema/migration changes are **ask-first** (`00-governance/agent-operating-rules.md` §2); applied migrations are on the never-touch list.
- An agent asked to "fix" a migration that already ran responds with a new forward migration, citing rule 2.

## Enforcement
- Mechanism: CI job
- Config: stacks/nextjs-default/ci/pr.yml (migrations job: any modified/deleted migration file fails the PR; only additions pass)
- Fallback if unenforceable: n/a — immutability is CI-enforced and the push ban is a never-touch rule; review quality of generated SQL is carried by the standing self-review items.

## Bootstrap
- What new-project.sh injects for this standard: the PR workflow containing the migration-discipline guard.
