# Security & Privacy Spine

Security is not a layer — it's a property every layer either upholds or leaks. This spine doesn't duplicate the layer rules; it shows **where security bites in each layer**, owns the cross-cutting rules no single layer can, and carries the data-privacy baseline.

## Where it bites, layer by layer

| 03 Design | Per-feature **threat model** before building qualifying features; error envelopes that leak no internals | `03-design/threat-model.template.md`, `03-design/api-contract-design.md` rule 5 |
| 04 Build | Boundary validation of all external input; **secrets never in git** (staged scan + history scan); boot-validated env | `04-build/secrets-config.md`, `04-build/coding-standards.md` rule 9 |

## Cross-cutting rules owned here

1. **Auth checks live at the resource, not the road.** Session/permission verification happens in the handler or server component that serves the data; middleware and route grouping are convenience layers, never the boundary. This is the durable principle behind governance guardrail 3 (CVE-2025-29927, the spoofable-middleware-header bypass — `00-governance/agent-operating-rules.md` §5): the specific CVE is patched, the design rule outlives it.
2. **Least privilege everywhere:** DB roles, API tokens, and host permissions get the minimum scope that works; the app's runtime credentials can't drop tables it only reads.
3. **Rotation:** any credential is rotatable without a deploy (env-injected, never baked into builds); rotate on schedule (yearly floor), on any suspicion, and **immediately** when a secret ever touched git (`04-build/secrets-config.md` rule 8).
4. **Dependencies are attack surface:** the allowlist, audit gate, and immediate security updates are the supply-chain posture; an agent may never add a dependency unilaterally.

## Data privacy & PII baseline

5. **Classify at design time:** every stored field is public / internal / PII / credentials-or-payment — the threat model template forces the question per feature.
6. **Minimize:** collect only fields with a named use; "might be useful" is not a use. Don't store what you can derive; don't retain what you no longer need (retention rides the backup ladder, `07-operations/backup-dr.md`).
7. **PII handling:** identify users by opaque ID in logs/analytics; PII never in URLs, error messages, or third-party telemetry; encrypted in transit always and at rest via the managed store's encryption.
8. **User rights:** deletion/export requests must be technically satisfiable — soft-delete and backup design account for erasure (hard-delete path exists; backups age out on the retention ladder).
9. **Breach reality:** if user data may have been exposed, the incident runbook's postmortem includes notification obligations assessment — decided with the law applicable to the project, not improvised.

## Cross-cutting rules owned here (cont.)

Concurrency and timing are correctness *and* security properties: a comparison that leaks via its runtime, or a write that two requests race, is exploited the same way a missing auth check is. These cross-cutting rules belong to the spine because no single layer owns them — they span the handler (04), the data model (03), and the operational retry path (06/07). Numbering continues past the data-privacy block to avoid collision.

10. **Constant-time comparison for secrets, tokens, and MAC/signature verification.** Equality of a secret value — session token, password-reset/API token, HMAC, webhook signature, CSRF token — is checked with a constant-time comparison, never an ordinary equality/short-circuit operator. Byte-by-byte equality that returns early on the first mismatch leaks how many leading bytes were correct through its timing, turning a brute-force search of the whole value into a per-byte one (a timing side channel). Hash the candidate and the expected value to a fixed length and compare in constant time, or use the platform's constant-time-equality primitive. The same rule covers verifying an inbound webhook/payment-callback signature before trusting its body. This is a per-diff judgment call, not a separate gate: it rides the **threat-model fallback already in `05-verification/code-review-standard.md`** — when a feature handles a secret/token/MAC, its threat model's Spoofing/Tampering rows must name constant-time verification as the mitigation, so the existing checklist line covers the review with no new §E item.

11. **State-changing writes that money/auth/inventory depend on declare their isolation level and guard against races.** A correctness-critical write — debiting a balance, decrementing stock, consuming a single-use token, enforcing a uniqueness invariant across rows — declares the transaction isolation level it requires and does not assume the engine's default suffices. The default (read-committed on most engines) permits **write-skew**: two transactions each read a consistent snapshot, each individually validates an invariant, and both commit, leaving the invariant violated (double-spend, oversold inventory, two rows where one was allowed). Use the strongest isolation the operation needs — serializable when the invariant spans rows the write doesn't itself touch — or take an explicit row/advisory lock (`SELECT ... FOR UPDATE` / equivalent) on the contended row, plus a database-level constraint as the backstop the application logic can't bypass. State the choice and its reason where the write lives; a money/auth/inventory write with an unstated isolation level is an unreviewed race. This concept's review also rides the existing threat-model fallback — the Tampering/Elevation rows of a money or inventory feature's model must name the isolation or locking strategy — so it adds no new §E checklist line.

12. **State-changing requests carry an idempotency key so a retry or double-submit can't double-apply.** Any request that creates or mutates state through a path a client (or a network) may legitimately repeat — payment capture, order placement, "send invite", an at-least-once webhook or queue delivery — accepts a client-supplied idempotency key, and the server records the key with the result of the first application so a replay returns that stored result instead of acting twice. A double-clicked submit button, a mobile client retrying on a flaky connection, and a webhook provider's at-least-once redelivery are the *expected* case, not the edge: without a dedup key they become a double charge, a duplicate order, or a doubled side effect. The key's uniqueness is enforced at the store (a unique constraint on the key column), pairing this rule with rule 11 — the constraint is the race backstop and the idempotency record in one. Review rides the threat-model fallback as well: a state-changing entry point's model must state how a replay of that entry point is made safe.

13. **The threat-model walk explicitly prompts for concurrent-access conflicts.** When `03-design/threat-modeling.md` walks STRIDE per element, a feature touching a shared mutable asset (balance, inventory, counter, single-use token, uniqueness invariant) asks the concurrency question as part of the Tampering and Elevation-of-privilege prompts: *what happens if two requests hit this asset at once, or the same request arrives twice?* The answer names the isolation/locking choice (rule 11) and the idempotency strategy (rule 12) as mitigations with a verification location, exactly as any other threat row does. This keeps concurrency safety inside the artifact the security pass already reviews — the per-feature `03-design/threat-model.template.md`, whose Threats and Mitigations tables carry these rows — rather than as a separate concern that needs its own gate. Because the review lands in the threat model, the existing threat-model fallback line in the self-review checklist already covers it; no new §E line is introduced.

## Standards basis

- **OWASP Top 10:2025** (owasp.org/Top10/2025) — the consensus most-critical web risks. This spine's posture maps directly: A01 Broken Access Control → rule 1 (auth at the resource) + rule 2 (least privilege); A02 Security Misconfiguration → boot-validated env, per-environment secrets; **A03 Software Supply Chain Failures** (new, elevated from 2021's A06) → rule 4 (deps as attack surface: allowlist + audit gate + immediate updates); A07 Authentication Failures → resource-level checks; A09 Security Logging and Alerting Failures → no secrets/PII in logs + observability; **A10 Mishandling of Exceptional Conditions** (new) → error envelopes that leak no internals.
- **OWASP ASVS 5.0** (May 2025, asvs.dev) — verification requirements by chapter (V2 Authentication, V4 Access Control, V6 Stored Cryptography, V8 Data Protection, V14 Configuration). The per-feature security acceptance criteria and the classify/minimize/PII rules below operationalize ASVS V8/V14 at design time.
- **OWASP Proactive Controls 2024** (top10proactive.owasp.org) — developer-facing techniques: C1 access control, C3 validate all input (boundary validation, layer 04), C2 cryptography, C8 secure secrets handling. Ordered controls, not just vulnerabilities.
- **OWASP SAMM** — security maturity model framing the threat-modeling-before-build cadence as a Design-domain practice, not an afterthought.
- **NIST SSDF (SP 800-218 v1.1)** — four practice groups PO/PS/PW/RV; this spine realizes PW.4 (boundary validation), PS.1 (secrets out of source), PW.7/PW.8 (review + test), and RV.* (CVE patching, layer 08). (SSDF v1.2 in public draft per EO 14306, Dec 2025 — not yet final.)
- **SLSA v1.0 Build Track** (slsa.dev/spec/v1.0) — provenance levels L1–L3 for supply-chain integrity; the supply-chain posture (rule 4) targets verifiable build provenance and tamper-resistance, aligning A03 / OWASP A08 (integrity).
- **CIS Benchmarks** — vendor-neutral secure-configuration baselines for hosts/containers/cloud; the least-privilege and managed-store-encryption rules track CIS hardening guidance.
- **GDPR Art. 5 principles** — data minimization (Art. 5(1)(c)), purpose limitation (Art. 5(1)(b)), storage limitation (Art. 5(1)(e)), integrity/confidentiality (Art. 5(1)(f)); these are the basis for rules 6–7. Privacy-by-design (GDPR Art. 25 / Cavoukian's 7 foundational principles) is why classification (rule 5) happens at design time, not retrofit. User rights (rule 8): GDPR Arts. 15/17 (access/export, erasure). Breach notification (rule 9): GDPR Arts. 33/34.
- **NIST Privacy Framework** (Identify-P / Govern-P / Control-P / Communicate-P / Protect-P) — CT.DM (Data Minimization) mandates collect-only-what's-needed, defined retention, scheduled deletion; this is the framework form of rules 6–8. (PFW 1.1 in initial public draft, Apr 2025; 1.0 remains current.)
- **OWASP Cryptographic Storage Cheat Sheet** and **OWASP Authentication Cheat Sheet** (cheatsheetseries.owasp.org) — both prescribe **constant-time comparison** for secrets, tokens, and MAC/HMAC verification to defeat timing side-channel attacks (the basis for rule 10); the Cryptographic Storage sheet also covers comparing fixed-length digests rather than raw values.
- **OWASP ASVS 5.0 V11 (Cryptography) and V6 (Stored Cryptography)** (asvs.dev) — verification requirements that comparisons of MACs/tokens/signatures be performed in constant time and that integrity/authenticity of inbound signed payloads (webhooks/callbacks) be verified before use; corroborates rule 10.
- **CWE-208 "Observable Timing Discrepancy"** and **CWE-385 "Covert Timing Channel"** (cwe.mitre.org) — the weakness classes a non-constant-time secret comparison realizes; rule 10 is the mitigation for both.
- **ANSI SQL transaction isolation levels** (READ UNCOMMITTED / READ COMMITTED / REPEATABLE READ / SERIALIZABLE) as documented in the **PostgreSQL "Transaction Isolation" chapter** (postgresql.org/docs/current/transaction-iso.html) — the canonical reference for the phenomena each level permits or prevents and for **`SELECT ... FOR UPDATE`** explicit row locking; direct basis for rule 11's "declare the level / take the lock."
- **"A Critique of ANSI SQL Isolation Levels"** (Berenson, Bernstein, Gray, Melton, O'Neil, O'Neil; Microsoft Research / SIGMOD 1995) — the paper that defines and names **write-skew** (anomaly A5B) and shows snapshot isolation does not prevent it; the literature basis for rule 11's double-spend / oversell / double-allocation hazard.
- **CWE-362 "Concurrent Execution using Shared Resource with Improper Synchronization (Race Condition)"** and **CWE-367 "Time-of-check Time-of-use (TOCTOU)"** (cwe.mitre.org) — the weakness classes rules 11–13 guard against when two requests race a shared mutable asset.
- **Stripe API — Idempotent Requests** (docs.stripe.com/api/idempotent_requests) — the established design pattern for rule 12: a client-supplied idempotency key stored with the first response so a retry returns the stored result instead of re-applying; the at-least-once-then-dedup model that webhook/queue redelivery also requires.

## Enforcement
- Mechanism: git hook
- Config: stacks/nextjs-default/hooks/pre-commit (staged secret scan) + stacks/nextjs-default/ci/pr.yml (secrets, deps jobs) — the mechanical subset; each layer row above names its own gate
- Fallback if unenforceable: n/a — mechanical pieces are gated in their layers; judgment pieces ride the threat-model and security fallback lines already in the self-review checklist.

- What new-project.sh injects for this standard: nothing additional — the hooks, CI jobs, env schema, and threat-model template it already injects are this spine's enforcement surface.
