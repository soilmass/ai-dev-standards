# Functional-Verification Spine

A green test suite is not a working product. "Tests pass" proves the *code* behaves under a runner; **functional verification** proves the *built, deployed artifact* actually works — it boots, serves every declared route, resolves the endpoints it declares (no dangling API path, no `report-uri` pointing at nothing), throws **zero console errors on load**, clears the automated a11y floor, and a primary user flow reaches its **expected outcome**, not merely "no 500". The gap between the two is where shipped-broken lives: a build that compiled and unit-passed but white-screens on hydration, a route the sitemap promises that 404s, a CTA that navigates into an error boundary, a CSP that silently blocks the very script the page needs. This spine doesn't duplicate the testing or CI layer rules; it shows **where functional verification bites in each layer**, owns the cross-cutting covenant no single test tier sees (the staged PR→preview→prod proof that the artifact-as-deployed works), and ties the smoke covenant — expected-outcome, build-health, endpoint-resolution, zero-console-errors — to the gate that runs it against the real preview deploy before promotion.

## Where it bites, layer by layer

| Layer | Functional-verification obligation | Lives in |
|---|---|---|
| 02 Product | Acceptance criteria are **Observable, Deterministic, Atomic** and each names the user-visible outcome to verify; an item is Done only when that outcome **holds end-to-end across all its merged PRs**, not when its unit tests go green | `02-product/acceptance-criteria.md`, `02-product/definition-of-ready-done.md` |
| 04 Build | The smoke covenant lives at the top of the test pyramid as the **real-browser, real-app** tier that asserts the critical path's expected outcome and zero console errors; failure-path coverage proves the named failure modes are provably testable, not aspirational | `04-build/testing-strategy.md` (E2E tier + § Failure-path coverage) |
| 05 Verification | The PR pipeline runs the smoke against the preview deploy and the per-route a11y/perf baseline against every sitemap route; an item is not Done until those gates are green on the **deployed** surface — that is the definition of done, not "the unit job passed" | `05-verification/ci-pipeline.md`, `05-verification/definition-of-done.md`, `05-verification/a11y-perf-gates.md` |
| 06 Delivery | The smoke is verified against the **preview deploy** (the exact build that promotes to production), through the standard PR→preview→prod flow with **no manual host-side step**; production gets the artifact that was previewed-and-smoked, not a rebuild | `06-delivery/deployment-strategy.md` |
| 07 Operations | Field truth: client error tracking and RUM catch the console error, the broken endpoint, and the failed flow that a real user actually hit — the production echo of the smoke covenant, closing the loop the lab gate's emulation can miss | `07-operations/observability.md`, `07-operations/product-analytics.md` |
| 08 Maintenance | A flaky or skipped smoke, a route quietly dropped from the smoke set, or an accepted "known console warning" is logged as functional-verification debt with a paydown trigger — never silently normalized | `08-maintenance/tech-debt-policy.md` |

## Cross-cutting rules owned here

1. **Verification is staged: PR → preview → prod, against the deployed artifact.** Functional verification runs not against a developer's local build but against the **preview deploy** — the exact build that promotes to production (`06-delivery/deployment-strategy.md`: one build previewed, then that build deployed from `main`). The smoke covenant is wired into the PR pipeline against that preview URL (`stacks/nextjs-default/ci/pr.yml`, job `smoke`) and the per-route a11y/perf baseline runs against the same surface; promotion happens only when both are green on what will actually ship. A unit suite passing against a local checkout tells you the code is sound; it does not tell you the deployed thing serves. This staging — verify the artifact-as-deployed, no manual host-side step between previewed and promoted — is the property no single test tier owns.
2. **The critical-path smoke proves the EXPECTED OUTCOME, not "no 500".** The smoke covenant (`stacks/nextjs-default/project-config/tests/smoke/critical-path.example.spec.ts`, run by the `smoke` Playwright project) asserts that a primary user flow reaches the state the user came for — the next step renders, the heading the success screen promises is visible, the form's confirmation appears — and not merely that the page returned a non-error status. A route can serve a 200 and still be functionally dead (a blank shell, an error boundary swallowing the failure, a CTA wired to nothing). "It returned 200" is the floor; "the user got what the flow promised" is the bar this spine sets.
3. **Build health and endpoint resolution: the artifact boots and everything it declares resolves.** The deployed build must boot and serve every route in the **smoke route set** (the routes the feature's acceptance criteria name, `02-product/acceptance-criteria.md`), and every endpoint the artifact *declares* must resolve — no dangling internal API route, no CSP `report-uri`/`report-to` aimed at a sink that 404s, no `<link>`/manifest/sitemap reference into the void. Internal page-link and sitemap resolution is owned site-wide by the design-system-coherence spine's site-integrity gate (`stacks/nextjs-default/project-config/scripts/check-site-integrity.example.sh`, `ALLOWED_404=0`); this spine adds the **boot-and-serve** proof on the deployed artifact for the smoke route set, so a build that compiles but white-screens, or serves home while a declared sub-route 500s, fails before promotion.
4. **Zero console errors on load.** Every smoke route must load with **zero** `console.error` and zero uncaught `pageerror` (calibrated budget **CAL-F02 = 0**, pinned identically in the smoke spec and the CI job env). A console error on load is the field-visible signature of a failed hydration, a rejected fetch, a missing asset, or a CSP violation — none of which a unit run against a local build surfaces, and all of which a real user's devtools (and your client error tracking, `07-operations/observability.md`) will. The budget is zero and stays zero: a genuine third-party-script error that cannot be silenced at its source is quarantined by stubbing/blocking that origin in the smoke run, never by lifting the budget. This complements the Lighthouse best-practices floor (`05-verification/a11y-perf-gates.md`, which also penalizes console errors) with a deterministic, expected-outcome-bearing assertion on the critical path.
5. **Named failure modes are provably testable, not aspirational.** A failure mode you can name — the upstream times out, the payment declines, the validation rejects, the session expires mid-flow — must have a test that exercises that exact path and asserts the **degraded-but-correct** behavior (`04-build/testing-strategy.md` § Failure-path coverage): the right error surface, the fallback, the safe state. "We handle errors gracefully" with no test that drives the error is a claim, not a verification. The smoke covenant carries the happy-path expected-outcome proof; the failure-path tier carries the named-failure proofs; together they make "it works" mean both "it does the thing" and "it fails the way we said it would."

## Standards basis

- **Google SRE — release engineering & canarying** (sre.google/sre-book/release-engineering/, sre.google/workbook/canarying-releases/) — verify a release against a representative surface before exposing it broadly; the basis for rule 1's PR-preview-as-pre-production-canary and the gate-before-promotion staging. The progressive-delivery extension (traffic-gated rollout) is owned by `06-delivery/deployment-strategy.md`; this spine owns the deployed-artifact smoke that precedes any promotion.
- **Continuous Delivery** (Humble & Farley) — every change is releasable through an automated pipeline that promotes *the same artifact* across environments rather than rebuilding it; the basis for rule 1's "verify the build that will actually ship, not a local rebuild" and for treating the preview deploy as the verification surface. The deployment standard (`06-delivery/deployment-strategy.md`) owns the promotion-flow half; this spine owns the proof-on-the-artifact half.
- **Smoke-testing best practice** (the build-verification test: a fast, shallow, broad check that the critical paths of a freshly built/deployed system work before deeper testing) — the basis for rules 2–4: shallow-but-real, expected-outcome-bearing, run against the deployed artifact, kept lean on the critical path. The breadth budget (≈20–30 revenue-critical E2E tests) and the flake-is-P1 hygiene that keep the smoke trustworthy are owned by `04-build/testing-strategy.md`.
- Builds on the testing and verification standards it ties together — `04-build/testing-strategy.md` (the E2E + failure-path tiers), `05-verification/ci-pipeline.md` and `05-verification/definition-of-done.md` (the pipeline that runs the smoke and the done-bar it feeds), `05-verification/a11y-perf-gates.md` (the per-route a11y/perf baseline on the same preview surface), `06-delivery/deployment-strategy.md` (the PR→preview→prod flow), and `07-operations/observability.md` (the field echo). It overlaps the design-system-coherence spine (`_spines/design-system-coherence.md`) on whole-site link/sitemap resolution and the reliability spine (`_spines/reliability.md`) on failure-path testing by design — those spines own the site-integrity-crawl and the resilience framing; this one owns the deployed-artifact functional smoke (boot, serve, resolve, zero-console-errors, expected-outcome) that proves the built thing works before it promotes.

## Enforcement
- Mechanism: CI job
- Config: stacks/nextjs-default/project-config/playwright.config.example.ts (the `smoke` Playwright project + the `SMOKE_MAX_CONSOLE_ERRORS = 0` budget, CAL-F02) + stacks/nextjs-default/project-config/tests/smoke/critical-path.example.spec.ts (the expected-outcome + zero-console-error covenant) + stacks/nextjs-default/ci/pr.yml (job `smoke`: runs the smoke project against the Vercel preview deploy before promotion) — the deployed-artifact functional gate. The adjacent gates this spine ties to keep their own configs: site-wide link/endpoint resolution rides stacks/nextjs-default/project-config/scripts/check-site-integrity.example.sh (`ALLOWED_404=0`, CAL-F01) and the per-route a11y/perf/best-practices baseline rides stacks/nextjs-default/ci/lighthouserc.json + stacks/nextjs-default/ci/pr.yml (lighthouse job).
- Fallback if unenforceable: n/a — boot/serve/resolve, the zero-console-error budget, and the primary-flow expected-outcome are smoke-gated against the preview deploy in CI; the judgment pieces (which routes belong in the smoke set, whether the asserted outcome is the *right* outcome, which named failure modes warrant a failure-path test) ride the acceptance-criteria, definition-of-done, and testing-strategy review items already aggregated in `05-verification/code-review-standard.md` §E (owned by `02-product/acceptance-criteria.md`, `02-product/definition-of-ready-done.md`, and `05-verification/load-testing.md`'s failure-path sibling in `04-build/testing-strategy.md`).

## Bootstrap
- What new-project.sh injects for this standard: nothing additional — the `smoke` Playwright project + the `SMOKE_MAX_CONSOLE_ERRORS` budget in the injected `playwright.config.ts`, the `tests/smoke/` covenant spec, and the `smoke` PR-pipeline job (run against the preview deploy) that the testing and CI standards inject are this spine's enforcement surface; the project fills in its real smoke route set and primary-flow assertions.
