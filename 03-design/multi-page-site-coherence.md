# Multi-Page Site Coherence

A site is a graph of pages, not a single landing page. Most front-end quality checks (a11y, performance, layout) bite **per page** and so silently pass a site whose pages are individually fine but **collectively broken** — a footer link that 404s, a nav that renders differently on the pricing page than on the home page, a sitemap that advertises a route the build never produced, a breadcrumb that points at a parent that does not exist. Those are the defects a per-page lens cannot see. This standard governs the **whole-site invariants**: the things that must hold *across* pages for the site to be coherent. The per-page bar (unique title/description, one `<h1>`, complete metadata, the a11y floor on every route) lives in `03-design/per-page-quality-baseline.md`; graphics/asset cohesion in `03-design/graphics-asset-cohesion.md`; the visual-layout rules in `03-design/responsive-layout-discipline.md`. This doc is the cross-page integrity layer those sit on top of.

The four invariants below are checked against the **built, served site** — the artifact a user actually loads — not the source. A link that resolves in a dev server but 404s in the production build is a shipped defect; the gate crawls the build.

## Link integrity — zero broken internal links

1. **Every internal link resolves in the built site. The allowed count of broken internal links is zero.** "Internal" is any link whose target is a route of this site: anchor `href`s, nav items, footer links, in-content cross-references, buttons that navigate, and the canonical/`og:url` targets. A link to a route the build did not produce is a 404 and a hard failure — there is no "minor" broken link, because a user who hits one has hit a dead end the site promised would work.
2. **Header, footer, and nav links are part of the contract, not decoration.** Shared chrome is the most-traversed surface on the site (it appears on every page), so a single broken footer link is a site-wide defect, not a one-page one. These links are crawled and resolved like any other.
3. **A link to an anchor (`/page#section`) resolves both halves:** the route exists *and* the named fragment corresponds to an element on that page. A fragment that scrolls nowhere is broken even though the route loads.
4. **External links are validated for shape, not liveness, at the gate.** An internal gate cannot own the uptime of a third-party domain, so external links are checked for well-formedness (scheme present, not a stray internal-looking relative path) and the liveness of external targets is a review/monitoring concern, not a build blocker. The hard zero in rule 1 is for *internal* links only.

## Shared chrome — identical across every page

5. **The header, the primary navigation, and the footer render identically on every page of the site.** Same items, same labels, same order, same destinations. A nav that lists six items on the home page and five on the pricing page is incoherent: the user's mental model of "where can I go" must not change as they move through the site. Shared chrome is authored once (a shared layout/component) and any per-page divergence is a defect unless it is a *declared, intentional* variation (see rule 7).
6. **The "current page" indication is the only sanctioned per-page difference in the nav.** Marking the active item (and the corresponding `aria-current`) legitimately differs page to page; the *set of items, their labels, and their targets* do not. The check compares the chrome's structure modulo the active-state marker.
7. **Intentional chrome variants are declared, not discovered.** Some sites legitimately vary chrome by section (a docs area with a sidebar the marketing pages lack; a checkout flow with a stripped header). Where that is intended, the variation is **declared** — the page or section states which chrome contract it follows — so the gate compares each page against the contract it claims rather than flagging every difference as drift. Undeclared chrome divergence is a defect; the absence of a declaration means "must match the site-wide chrome."

## Sitemap ↔ routes ↔ links coherence

8. **The sitemap, the set of actually-built routes, and the internal links are mutually coherent — three views of one route set that must agree.** Specifically: every route the sitemap advertises is a route the build produced (no sitemap entry points at a 404); every indexable, linkable route the build produced appears in the sitemap (no orphan route silently excluded from discovery); and every internal link target is a real route. A sitemap that lists a page that does not exist misleads crawlers and users alike; a real page absent from the sitemap is invisible to search.
9. **Routes deliberately excluded from the sitemap are excluded deliberately.** Utility routes (auth callbacks, raw API endpoints, intentionally `noindex` pages) are legitimately absent from the sitemap — but the exclusion is a stated intent (a `noindex`/robots signal or an explicit exclude list), not an accident. The coherence check treats a declared-excluded route as coherent and an *undeclared* missing route as a finding.
10. **One canonical route per page of content.** Where multiple paths can reach the same content, exactly one is canonical and the others redirect or carry a canonical reference to it; the sitemap lists the canonical form. Duplicate routes serving the same content with no canonical winner is an incoherence the gate flags.

## Breadcrumb and parent-reference validity

11. **Every breadcrumb segment and every "parent"/"up" reference points at a page that exists.** A breadcrumb is a promise about the site's hierarchy; a segment that links to a missing parent breaks both the navigation and the structured-data it often emits. Parent references are resolved against the built route set exactly like any other internal link (rule 1).
12. **The breadcrumb hierarchy is consistent with the route structure and the chrome.** The parent a page claims via its breadcrumb is the same parent its position in the nav/route tree implies — a page cannot breadcrumb to one parent while the nav nests it under another. Where breadcrumbs emit structured data (the `BreadcrumbList` schema), the emitted hierarchy matches the on-page one.

## How the gate reads these rules

13. The four invariants are mechanically checkable against a built, served site, and that is where they are enforced: the preset ships a **built-site integrity crawler** (`check-site-integrity`) that crawls every route, resolves every internal/nav/footer link and fragment, diffs the shared chrome across pages against each page's declared contract, reconciles the sitemap against the produced routes and the link graph, and validates breadcrumb/parent references. The gate runs in CI against the production build (per-PR and nightly), so an incoherence is a failed check, not a post-launch discovery. The threshold (zero broken internal links) and the crawler's configuration are owned by the preset gate and tracked as its calibration knob; this doc states the rules the gate enforces.

## Standards basis

- **Web Content Accessibility Guidelines (WCAG) 2.2** — *Guideline 2.4 Navigable* and **3.2.3 Consistent Navigation (Level AA)**: navigational mechanisms repeated across pages occur in the same relative order each time they are repeated. This is the conformance basis for the shared-chrome invariant (rules 5–7) — consistent navigation is an accessibility requirement, not only an aesthetic preference (w3.org/TR/WCAG22/, w3.org/WAI/WCAG22/Understanding/consistent-navigation.html).
- **Sitemaps XML protocol** (sitemaps.org): a sitemap lists the URLs a site wants discovered and is a hint to crawlers about the canonical set of pages. Grounds the sitemap↔routes coherence rule (8–9) — the sitemap must describe the routes that exist, and routes meant to be discoverable must appear in it.
- **Google Search Central — Crawling & Indexing / Canonicalization & Sitemaps best practices** (developers.google.com/search): broken internal links waste crawl budget and dead-end users; sitemaps should list canonical, indexable URLs only; duplicate URLs need a single declared canonical. Grounds rules 1, 8, and 10 (one canonical route per page, sitemap lists canonical forms).
- **Breadcrumb structured data — schema.org `BreadcrumbList`** (schema.org/BreadcrumbList; Google's breadcrumb guidance): breadcrumbs express a page's position in the site hierarchy and, when emitted as structured data, must mirror the on-page trail and reference real pages. Grounds the breadcrumb-validity rules (11–12).
- **Information architecture — site as a navigable graph** (Rosenfeld, Morville & Arango, *Information Architecture for the Web and Beyond*): a site's coherence is a property of the relationships *between* pages — navigation, hierarchy, and labeling consistency — not of any page in isolation. The conceptual basis for treating cross-page integrity as a first-class standard distinct from per-page quality.
- **HTTP semantics — 404 Not Found** (RFC 9110 §15.5.5): a request for a target resource the server does not have is a 404; a link whose target returns 404 is by definition broken. Grounds the "zero broken internal links" definition (rule 1) against the served artifact rather than the source.

## Enforcement
- Mechanism: CI job
- Config: stacks/nextjs-default/project-config/scripts/check-site-integrity.sh (built-site integrity crawler — link/404 resolution, shared-chrome diff, sitemap↔routes reconciliation, breadcrumb-parent validation; wired into the PR and nightly workflows). The zero-broken-internal-links threshold and crawler configuration are the gate's calibration knob, registered with that gate.
- Fallback if unenforceable: n/a — the four cross-page invariants are mechanically checkable against the built, served site and are gated by the `check-site-integrity` crawler; external-link liveness (deliberately out of the internal hard-zero per rule 4) rides the monitoring/review concern named in that rule, not this standard.

## Bootstrap
- What new-project.sh injects for this standard: nothing of its own — the standard is enforced by the `check-site-integrity` gate the preset injects (the script plus its wiring into the PR and nightly workflows); this doc is the rule set that gate enforces.
